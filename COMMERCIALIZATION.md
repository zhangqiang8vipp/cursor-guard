# Cursor Guard — 商业化规划

> 本文档描述 cursor-guard 的商业化边界、阶段性策略与落地节奏。
> 它回答的不是“功能怎么做”，而是“这套东西将来怎么活下去”。
>
> **当前阶段**：开源项目 / 标准探索期  
> **当前结论**：先做强 OSS 与协议，不提前承诺自己接不住的企业责任

---

## 一句话结论

cursor-guard 当前不是“企业级支持产品”，而是一个**开放的 AI Code Safety 项目**。

短期目标：

- 把产品做稳
- 把协议讲清楚
- 把用户做起来
- 把行业心智占住

长期目标：

- 如果未来真的具备支持能力，再把“治理层”做成企业可采购的官方方案

---

## 商业化原则

| 原则 | 含义 |
|---|---|
| 个人开发者永远优先 | 不靠卡个人用户功能赚钱 |
| 核心能力优先开放 | 安全网、协议、参考实现优先扩散，而不是优先收费 |
| 不承诺接不住的责任 | 现在不卖 SLA，不卖“有人负责”，不卖合规背书 |
| 企业付费只能买“更省事” | 企业若付费，买的是治理成本下降、支持、定制与组织能力，不是基础可用性 |
| 协议价值高于闭源价值 | 能成为事实标准，比早期收窄许可更重要 |
| 先验证需求，再决定收费形态 | 不提前写死价格、许可证和企业版边界 |

## 先回答一个现实问题

### 企业可以直接拿开源版商业使用吗？

可以，而且这是正常情况。

企业完全可能：

- 使用开源版
- 自己改造
- 自己部署
- 自己做内部治理能力

cursor-guard 的商业化前提，不是“阻止企业用开源版”，而是：

> **让企业意识到：自己维护整套治理能力，通常比直接采购官方方案更贵、更慢、更不稳定。**

所以商业化的重点，不是“防白嫖”，而是：

- 提供更完整的组织能力
- 提供更低的治理成本
- 提供更省心的落地方式
- 提供更可信的官方实现

---

## 当前阶段的真实定位

### 现在不卖什么

当前阶段，cursor-guard 明确**不卖**以下东西：

- 不卖“AI 一定安全”
- 不卖“出了事我们负责”
- 不卖“审计一定通过”
- 不卖“企业支持 / SLA”
- 不卖“合规背书”

原因很简单：

- 这些承诺需要稳定的团队、支持体系、运维能力、法务能力
- 当前项目还不应该承担这些重量级责任

### 现在卖什么

严格说，**当前阶段不以产品功能收费为主**。

如果一定要有商业动作，当前最适合卖的是：

- 顾问式接入建议
- 定制化 Policy Pack 设计
- 团队培训
- 有边界的集成支持

也就是：

> **先卖时间和专业判断，不卖企业级承诺。**

## 商业化阶段划分

### 阶段 0：OSS 与标准探索期（现在）

### 目标

- 把 `V2-V3` 做稳
- 建立用户口碑
- 验证真实使用场景
- 明确协议雏形

### 交付重点

- 开源核心能力持续可用
- 文档、恢复路径、doctor、自动备份体验打磨到位
- `ROADMAP` 讲清楚“为什么这不是普通备份工具”

### 商业动作

- 不做正式企业版
- 不做 License Key
- 不做收费门槛设计
- 不承诺 SLA

### 成功标志

- 有真实用户持续使用
- 有用户愿意主动推荐
- 有人开始把 cursor-guard 当成“AI code safety”工具提起

---

### 阶段 1：设计伙伴期（建议在 V3-V4 之间启动）

### 目标

- 找到 5 到 10 个真实团队
- 验证哪些问题他们真愿意付费解决
- 用最小成本测试“企业价值”是否成立

### 这阶段卖什么

不是卖“企业版功能”，而是卖：

- 接入辅导
- 团队实践建议
- 内部流程梳理
- 定制 Policy Pack 草案
- 有边界的集成支持

### 这阶段不卖什么

- 不卖标准化企业版套餐
- 不卖正式 SLA
- 不卖“合规保证”
- 不卖 7x24 支持

### 为什么这一步重要

因为你现在最缺的不是定价表，而是：

- 企业到底在乎什么
- 企业是否真愿意为“治理层”买单
- 企业要买的是功能、支持、还是咨询

---

### 阶段 2：有限商业化（只有条件满足才进入）

### 进入条件

只有同时满足下面这些条件，才建议进入正式商业化：

- `V3-V4` 已经稳定
- 至少有数个真实团队在长期使用
- 设计伙伴已验证有重复出现的付费需求
- 已能稳定提供有限支持
- 已经清楚哪些能力属于“官方治理方案”，哪些必须保持开放

### 这阶段可以卖什么

可以开始卖：

- 官方支持
- 团队级 Policy Packs
- 审计导出
- CI 安全覆盖率报表
- Attestation 工具链
- 私有部署协助

### 更准确的定位

这时卖的不是“更高级的备份工具”，而是：

> **AI 编辑安全治理套件**

企业购买的价值是：

- 更低的治理成本
- 更低的内部落地成本
- 更强的审计就绪能力
- 更可靠的官方实现

---

### 阶段 3：企业方案成立后（远期）

如果将来真的走到这一步，商业模式应当是：

| 层级 | 内容 | 定位 |
|---|---|---|
| `OSS / Free` | 开源核心、协议、参考实现、基础恢复与诊断能力 | 负责扩散标准、建立默认心智 |
| `Official Governance Package` | Attestation、签名审计链、团队策略、CI 治理、官方支持 | 负责服务组织级落地 |
| `Professional Services` | 培训、定制集成、策略共建、迁移辅导 | 负责高客单价深度合作 |

注意：

- 这里依然不是“把基础能力锁起来”
- 而是“让组织能力、治理能力、支持能力成为可采购项”

---

## 什么东西应该保持开源

以下内容越开放，越有利于 cursor-guard 形成标准地位：

| 内容 | 建议 |
|---|---|
| 安全协议文档 | 开放 |
| Core 基础能力 | 开放 |
| 本地快照 / 恢复能力 | 开放 |
| MCP 基础工具接口 | 开放 |
| conformance suite | 开放 |
| 基础 doctor / 状态检查 | 开放 |

原因：

- 这些能力越开放，越容易成为事实标准
- 越多人实现这套协议，cursor-guard 的心智越强

---

## 哪些东西未来可以商业化

如果以后真的进入企业方案阶段，更适合收费的是：

| 能力 | 为什么适合收费 |
|---|---|
| 团队级 Policy Packs | 属于组织治理，不是个人基础能力 |
| Attestation 导出与验证工具 | 更接近企业交付物 |
| 签名审计链增强版 | 更偏团队级可信证据 |
| CI 安全覆盖率报表 | 更偏团队治理和审计准备 |
| 私有部署支持 | 属于服务与交付能力 |
| 官方支持 / 培训 / 定制集成 | 本质是时间、经验和响应能力 |

一句话：

> **适合收费的，不是“安全网本身”，而是“把安全网变成组织治理能力”的那部分。**

---

## 现在不该做的商业化动作

为了不把项目带偏，当前阶段明确不建议做：

- 提前做复杂的 License Key 系统
- 提前设计精细价格表
- 提前承诺 Enterprise SLA
- 提前宣传“已适合企业采购”
- 为了收费而把开源核心故意做残
- 把“协议开放”改成 source-available 只为防白嫖

这些动作最大的问题不是“技术难”，而是：

> **会把项目从“建立标准和信任”带偏到“提前卖货”，而你现在最重要的恰恰不是卖货。**

---

## 未来如果要收费，应该怎么说

最稳的外部口径不是：

- “我们保证 AI 不出事”
- “我们保证审计通过”
- “我们保证企业合规”

而是：

> **cursor-guard 帮助团队把 AI 编辑安全流程记录成可验证证据，并以更低的治理成本完成内部落地。**

也可以更直白一点：

> **企业购买的不是绝对安全，而是更低的治理成本、更强的审计就绪能力，以及更可验证的流程证据。**

---

## 接下来 12 个月最合理的动作

| 优先级 | 动作 | 目标 |
|---|---|---|
| `P1` | 把 `V2-V3` 做到足够稳 | 让核心产品可信 |
| `P1` | 继续完善文档、协议表达、恢复体验 | 让外界看懂这是什么 |
| `P1` | 建立用户反馈渠道，记录真实痛点 | 确认市场信号 |
| `P2` | 找少量设计伙伴团队做深度访谈 | 验证企业真正愿意付钱的点 |
| `P2` | 明确“开放能力”与“治理能力”的边界 | 为未来商业化留结构 |
| `P3` | 如果设计伙伴需求成立，再考虑官方治理包雏形 | 进入有限商业化准备 |

---

## 最终判断

cursor-guard 的商业化不应该从“怎么拦住企业白嫖”开始，而应该从“怎样成为 AI Code Safety 的默认参考实现”开始。

只有先建立：

- 用户信任
- 协议心智
- 开源影响力
- 真实使用场景

未来的商业化才会顺。

所以这份规划的结论很简单：

> **现在先做强 OSS 与标准。**  
> **以后如果真的有能力接住，再把治理层做成官方可采购方案。**
